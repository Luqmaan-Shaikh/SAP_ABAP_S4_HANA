## ABAP PROXY :-

-> In SAP ABAP, inbound delivery and outbound delivery proxies refer to the asynchronous communication interfaces used to exchange delivery data between SAP and external systems (like warehouse logistics providers, or other SAP systems) using proxies (usually created from Enterprise Services or via SOA tools like SPROXY)

# What is a Proxy in SAP ABAP :-

-> A Proxy is an ABAP object generated from an Enterprise Service Definition (WSDL/XML schema) 
-> It allows the SAP system to send or receive messages in XML format, typically used in SAP PI/PO, SAP Cloud Integration, or other middleware scenarios.

# Outbound Delivery :-

- Meaning :
-> An outbound delivery proxy sends delivery data from SAP to an external system

- Use Case :
-> Notify a warehouse system about deliveries 
-> Send outbound delivery details (items, quantities, shipping info)

- Flow :
-> Delivery is created or changed (ex: in VL01N, VL02N)
-> ABAP Program or output type triggers the proxy class

-> The Proxy sends XML data to the middleware or external system

# Inbound Delivery :-

- Meaning :
-> An Inbound Delivery proxy is used when SAP receives delivery data from an external system

- Use Case :
-> A 3PL (third-party logistics) system sends inbound delivery confirmation 
-> ASN (Advanced Shipping Notification) Integration

- Flow :
-> External System sends delivery data (in XML)
-> Middleware routes it to SAP
-> SAP uses the inbound proxy class to receive and process the data (ex: create/update delivery document in SAP)

- Technical Tools Involved :
-> SPROXY - To view or generate proxy classes
-> SOAMANAGER - To configure endpoints (for web services)
-> PI/PO/SAP Cloud Integration - Middleware for routing messages 

- Summary :

Proxy Type		Description	Purpose					Triggered by

Outbound Delivery - SAP -> External 	Send Delivery info (EX: Shipping) 	ABAP, Output

Inbound Delivery - External -> SAP 	Receives Delivery Data (Ex: ASN)	Middleware Call

## Diff btw Inbound and Outbound Delivery Proxy :-


-				Outbound Delivery 					VS  		Inbound Delivery

1. Data Direction - From SAP to External System 					VS 	From External System to SAP
2. Trigger Point - ABAP program, user actions, or batch job 				VS 	External mssg via PI/PO or web service
3. Generated Class Role - Client Side Proxy 						VS 	Server Side Proxy
4. Call Method - Initiated by SAP 							VS 	Involved automatically by runtime engine
5. Customer Use Case - Sales Org, Delivery Updates, Material Master Replication 	VS 	Purchase Order Confirmation , status updates
6. Technical Term - 'Class Proxy' 							VS 	'Server Proxy'


-> SAP ABAP, SPROXY is a transaction code used to mange and generate ABAP proxies, which are used for commn with other systems, particularly via SAP Process Integration PI or SAP Process Orchestration PO
-> These Proxies act as interfaces, enabling data exchange btw SAP Systems and External Systems using standard protocols like SOAP and HTTP

## CDS Join Types :-

=> CDS Join Types with Examples:

1. Inner Join
->An Inner Join combines records from two or more tables only when there is a match in both tables. If there is no match, the rows are not included in the result set. It is one of the most commonly used join types and works well when you need data that is present in both tables.

- Cardinality:
-> Ø 1:1 or 1: N: Depending on the relationships between the tables, an inner join can have a one-to-one or one-to-many relationship.

- Use Case:
-> An Inner Join is useful when you only want records that have related data in both tables. For example, if we want to get purchase order details (from EKKO and EKPO) for items that are actually ordered, an inner join would be the appropriate choice.

- Example:
-> Let’s say we want to get purchase order details (EKKO) along with the items (EKPO) from a specific company code.

2. Left Outer Join
-> Retrieves all rows from the left table and matching rows from the right table. If no match exists, the result includes null values for the right table.

- Cardinality and Behavior:
-> 1:1: Keeps unmatched rows with null values.
-> 1: N: Ensures all rows from the left table appear, even if the right table has no match.
-> N:M: Needs careful design to avoid unnecessary data duplication.

- Use Case:
-> Fetch all deliveries from LIKP, including their corresponding items in LIPS, even if items are missing.
-> Retaining Non-Matching Records from Left Table

3. Right Outer Join:
-> The Right Outer Join is similar to a Left Outer Join but focuses on retaining all rows from the right table.
-> Retrieves all rows from the right table and matching rows from the left table. If no match exists, the result includes null values for the left table.

- Cardinality and Behavior:
1. 1:1: Returns rows even if the left table has no match.
2. 1: N: All rows from the right table appear, matched with rows from the left.

- Use Case:
-> Combine open items with cleared items, retaining all cleared items with BSID, BSAD Tables.

## ABAP RESTful Application Programming :- 

1. Data Modelling and Behaviour 
. the data model is collection of different entities involved in a business scenario, for example airlines, flight routes and flight schedules, and their relationships, for example the parent-child relationship between airline and the flight schedule.
. RAP uses CDS to define and organize the data model.
. the behaviour describes what can be done with the data model, for example if the data can be created, updated or deleted.
. the behaviour can be implemented using ABAP.

2. Business Service
. service definition is used to define what part of the CDS / Behavior definition will be exposed to next level.
. service binding defines a specific communication protocol such as OData V2 / OData V4.

3. Service Consumption
. service can be exposed as a UI Service- this is consumed by SAP Fiori Elements app.
. an OData service that is exposed as a Web API comes without any UI specific information in the metadata. it is the public interface for any OData client to access the OData service. for example, you can consume a Web API from another OData service.
